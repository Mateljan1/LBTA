LBTA City Payment Flow - Simple Explanation

This is the exact system now in place so leads do not fall through the cracks between website form submission and City payment.

What problem this solves

Before: someone could submit on LBTA site, but we had weak visibility into whether they actually finished payment in City Rec1.

Now: every registration-related lead gets a workflow status, stale leads are detectable, and City-paid records can be reconciled back into LBTA data.

Core statuses now used

pending_city_payment Lead submitted intent, but city payment is still outstanding

city_paid We confirmed the family completed payment in City

not_required City payment does not apply (example: private inquiry, racquet rescue)

Flow (in plain English)

1

Lead submits on LBTA website

From trial/register pages (or registration-assist modal), the lead is saved with a workflow state.

2

Lead is marked as pending city payment

Registration-related sources default to pending_city_payment so we can track unresolved intent.

3

Stale-lead checks find people stuck in limbo

We can run stale reports locally or by cron endpoint to list leads still pending after a time threshold.

4

Ops reconciles city-paid families

Using email list from City export, we dry-run then update lead workflow to city_paid.

5

Daily KPI summary gives visibility

We can pull one summary showing new leads, trials, pending 24h/72h, and city-paid counts.

What was added today

  • City payment sync logic (dry-run + real update mode).

Commands you can run

  • npm run leads:stale-city-payment
  • DRY_RUN=1 npm run leads:city-payment-sync -- family@example.com
  • npm run leads:city-payment-sync -- family@example.com
